The `actions/setup-java` v5.5.0 release adds signature verification for downloaded
JDKs, support for a new distribution, and several quality-of-life improvements for
Maven users. Here's what's new since v5.4.0.

## Verify JDK download signatures

You can now have the action cryptographically verify the JDK archive it downloads
before installing it. Set `verify-signature: true` and, when the distribution
supports it, the action downloads the detached GPG signature and validates the
package against the distribution's public key.

```yaml
- uses: actions/setup-java@v5
  with:
    distribution: temurin
    java-version: '21'
    verify-signature: true
```

Signature verification is supported today for the **Temurin** and **Microsoft**
distributions. If you enable it for a distribution that doesn't support it, the
action fails fast rather than silently skipping the check. You can also supply your
own trusted key with `verify-signature-public-key` (an ASCII-armored GPG public key)
to override the bundled default.

## New distribution: Tencent Kona JDK

`kona` joins the list of supported distributions, so you can install
[Tencent Kona JDK](https://tencent.github.io/konajdk/) directly:

```yaml
- uses: actions/setup-java@v5
  with:
    distribution: kona
    java-version: '21'
```

## Install a JDK without making it the default

The new `set-default` input lets you install a JDK **without** changing the job's
default Java. This is useful when a custom action or a single step needs a specific
JDK, but you don't want to disturb the Java version the rest of your workflow relies on.

```yaml
- uses: actions/setup-java@v5
  with:
    distribution: temurin
    java-version: '21'
    set-default: false
```

When `set-default: false`, `JAVA_HOME` and `PATH` are left untouched, but
`JAVA_HOME_<major>_<arch>` is still exported so the JDK stays discoverable, and the
JDK is still registered in Maven toolchains. `set-default` defaults to `true`, so
existing workflows keep their current behavior.

## Cleaner Maven build logs by default

Maven's artifact transfer progress can flood CI logs with hundreds of download
lines. The action now sets `-ntp` (`--no-transfer-progress`) in `MAVEN_ARGS` by
default for Maven 3.9+ and the Maven Wrapper, giving you quieter, more readable logs
out of the box. Any existing `MAVEN_ARGS` value is preserved. Want the progress output
back? Set `show-download-progress: true`.

Additionally, the `settings.xml` generated by the action now sets
`<interactiveMode>false</interactiveMode>`, so Maven never blocks a CI run waiting on
an interactive prompt.

## Fix: Maven toolchains no longer grow unexpectedly

Running the action multiple times in a job (for example, to register several JDKs in
`~/.m2/toolchains.xml`) previously appended duplicate entries. The generated
`toolchains.xml` is now de-duplicated by toolchain type and id, and existing root
attributes and non-JDK toolchains you manage yourself are preserved.

## Previously in v5.4.0

The v5.4.0 release shipped without a changelog post, so a few notable additions are
worth calling out here:

- **GraalVM Community distribution.** Alongside the existing Oracle `graalvm`
  distribution, you can now install the free GraalVM Community edition with
  `distribution: graalvm-community`.

  ```yaml
  - uses: actions/setup-java@v5
    with:
      distribution: graalvm-community
      java-version: '21'
  ```

- **`javac` problem matcher.** The action now registers a problem matcher, so Java
  compiler errors and warnings surface as inline annotations on your pull requests and
  commits instead of being buried in the build log.

- **Maven Wrapper caching.** When you enable `cache: maven`, the action now also caches
  the downloaded Maven Wrapper distribution (`~/.m2/wrapper`), keyed on
  `**/.mvn/wrapper/maven-wrapper.properties` in addition to `**/pom.xml`, for faster
  wrapper-based builds.

## Get started

Bump your workflows to the latest major tag to pick up these changes:

```yaml
- uses: actions/setup-java@v5
```

For the full list of inputs and examples, see the
[README](https://github.com/actions/setup-java#readme) and
[advanced usage guide](https://github.com/actions/setup-java/blob/main/docs/advanced-usage.md).
As always, we welcome feedback and issues in the
[actions/setup-java](https://github.com/actions/setup-java/issues) repository.
